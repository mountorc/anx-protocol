# API 类型

## 概述

ANX 系统支持两种 API 模式，用于不同场景下的页面内容获取。

## 模式一：配置型 API

### 描述

提供页面 ANX 配置的 API，加载接口时会返回页面对应的配置，当中不包含需要动态加载的数据。

### 工作流程

1. 客户端请求 API 获取页面配置
2. API 返回页面的 ANX 配置（不包含动态数据）
3. 客户端将配置传递给 anxcore 进行执行处理
4. anxcore 根据配置加载动态数据
5. 最终生成完整的页面内容

### 适用场景

- 客户端需要灵活控制数据加载过程
- 网络环境较好，客户端有足够的处理能力
- 需要在客户端进行配置的动态修改或扩展

### 示例响应

```json
{
  "config": {
    "page": "home",
    "components": [
      {
        "type": "list",
        "dataSource": "/api/data/list",
        "template": "list-item"
      }
    ]
  }
}
```

## 模式二：完整信息型 API

### 描述

提供页面完整信息的 API，直接返回包含完整的动态数据的页面内容。

### 工作流程

1. 客户端请求 API 获取完整页面信息
2. 服务器端先安装 anxcore-sdk
3. API 请求时会先加载配置到 anxcore-sdk 中
4. 使用 SDK 进行动态数据的加载
5. 整合为完整页面后返回给用户

### 适用场景

- 客户端处理能力有限
- 需要减少客户端请求次数
- 服务器端需要对数据进行预处理或权限控制
- 对页面加载速度要求较高

### 示例响应

```json
{
  "page": "home",
  "content": {
    "components": [
      {
        "type": "list",
        "data": [
          { "id": 1, "title": "Item 1" },
          { "id": 2, "title": "Item 2" }
        ],
        "template": "list-item"
      }
    ]
  }
}
```

## 两种模式的对比

| 特性 | 配置型 API | 完整信息型 API |
|------|------------|----------------|
| 响应内容 | 仅配置，不含动态数据 | 完整内容，包含动态数据 |
| 客户端处理 | 需要 anxcore 执行 | 直接使用响应内容 |
| 服务器端依赖 | 无特殊依赖 | 需要安装 anxcore-sdk |
| 网络请求 | 可能需要多次请求（配置 + 数据） | 单次请求 |
| 灵活性 | 高 | 低 |
| 性能 | 客户端处理开销较大 | 服务器端处理开销较大 |

## 选择建议

- 对于复杂页面且客户端性能较好的场景，建议使用配置型 API
- 对于简单页面或客户端性能有限的场景，建议使用完整信息型 API
- 可根据具体业务需求和技术架构选择合适的 API 模式
